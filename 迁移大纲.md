# simple-nextjs-jwt 业务迁移大纲

## 1. 迁移目标与范围
- 源项目：`D:/frontend/main/nextnestnuxt/simple-nextjs-jwt`
- 目标项目：`D:/frontend/main/1.operations/taxonomy`
- 目标：将源项目中的“业务能力”迁入目标项目，保持核心流程可用，并对齐当前技术栈（Next.js + NextAuth + Prisma + Contentlayer）。
- 非目标（首阶段不做）：
  - 完全一致的 UI 像素级还原
  - 一次性重构所有历史代码
  - 无回归前提下的大规模性能优化

## 2. 迁移原则
- 先跑通主链路，再补边角能力。
- 先兼容，后重构（避免“边迁边大改”）。
- 每个阶段可独立验收、可回滚。
- 迁移过程全程保留映射清单与变更记录。

## 3. 现状盘点（Inventory）
### 3.1 模块盘点
- 路由页面（公开页/登录页/业务页）
- API 接口（入参、鉴权、响应结构、错误码）
- 认证授权（JWT 生成、刷新、权限校验）
- 数据模型（表结构、索引、约束、枚举）
- 中间件（鉴权、日志、限流、跨域）
- 配置与环境变量（本地/测试/生产）

### 3.2 依赖盘点
- npm 依赖与版本
- 第三方服务（邮件、支付、对象存储、OAuth）
- 脚本（构建、代码生成、数据初始化、部署）

### 3.3 输出物
- 《源项目能力清单》
- 《接口清单与调用关系图》
- 《数据模型对照清单》

## 6. 测试策略
- 单元测试：关键工具函数、权限判断逻辑
- 集成测试：API + DB + Auth 关键路径
- E2E 冒烟：注册/登录/核心业务流
- 回归基线：迁移前后接口行为对比（状态码、字段、异常）

## 7. 风险与应对
- 风险：数据不一致  
  应对：双写校验脚本 + 抽样核对 + 回滚脚本

- 风险：鉴权逻辑偏差  
  应对：权限矩阵测试 + 中间件灰度开关

- 风险：环境变量缺失/格式错误  
  应对：`env.mjs` 强校验 + `.env.example` 同步更新

- 风险：第三方依赖网络/限流问题  
  应对：重试机制、超时配置、降级策略
